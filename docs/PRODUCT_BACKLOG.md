# Product Backlog — python-blackjack

## How to Use This Backlog

- PBIs are numbered sequentially: PBI-1.1, PBI-1.2, PBI-2.1, etc.
- The first number is the epic; the second is the item within the epic.
- Adam picks the next PBI and hands it to Crog via a task prompt.
- Completed PBIs are marked ✅ and recorded in CHANGELOG.md.

## Epic 2 — The Arena

The Arena transforms python-blackjack from a single-player simulator into a competitive platform where any combination of player types can face off at the same table. Players may be AI-backed (any provider), human (interactive CLI), or strategy bots (deterministic or stochastic). All hands are logged to JSONL as normal. Post-match analysis is generated by the AI viewer.

| ID | Description | Status |
|---|---|---|
| PBI-2.1 | Pluggable AI provider infrastructure — `AIProvider` protocol in `src/providers/`, concrete implementations for Anthropic, OpenAI, Grok, Ollama, and Mock (testing). Provider selected via `--provider` CLI flag or `AI_PROVIDER` env var (default provider — tournaments override per-player via `provider:name` syntax in `--players` (e.g. `anthropic:claude`, `openai:gpt-4o`)). Provider-specific config (model name, base URL, API key) via env vars. All implementations share identical interface: `query(context: str, prompt: str) -> str`. | Not started |
| PBI-2.2 | AI player strategy — `AIStrategyProvider` wraps any `AIProvider` and implements the `Callable[[Hand], str]` strategy interface. Describes hand state to the model, parses "hit" or "stand" from response. Note: dealer upcard not available at call time under current `Callable[[Hand], str]` interface — signature extension deferred to PBI-2.2 implementation; align with ICE-6 if upcard context is needed. Pluggable at `Player` construction. Requires PBI-2.1. | Not started |
| PBI-2.3 | Stochastic strategy support — add stochastic strategy implementations with probability-weighted policies. A stochastic strategy is defined by a fixed policy (e.g. "hit on 15 with p=0.5") with randomised execution. Seeded for reproducibility in tests. Built-in stochastic strategies: `WeightedStrategy` (per-hand-value hit probability table), `NoisyDealerStrategy` (dealer mirror with configurable error rate). All still implement `Callable[[Hand], str]`. | Not started |
| PBI-2.4 | AI viewer mode — `--ai` flag on `src/viewer.py` opens an interactive query loop. Loads and parses JSONL files as today, serialises events into prompt context, accepts natural language questions, streams responses to stdout. Requires PBI-2.1. | Not started |
| PBI-2.5 | Tournament runner — `src/tournament.py` orchestrates multi-table sessions with mixed player types. CLI: `python -m src.tournament --players anthropic:claude,openai:gpt-4o,human:adam,bot:dealer-mirror --hands 100 --tables 5`. Each player backed by a named provider or strategy. All hands logged to JSONL as normal. Requires PBI-2.2 and PBI-2.3. | Not started |
| PBI-2.6 | Post-match analysis — `--analyse` flag on viewer AI mode generates a narrative tournament report from the event log. `--perspective each` generates individual post-match reflections written from each player's point of view. ## Icebox

| ID | Description |
|---|---|
| ICE-1 | Card counting strategy — pluggable card behaviour (hit/stand based on running count) and pluggable bet behaviour (bet size based on running count). Requires extending the strategy interface to take deck state as input. |
| ICE-2 | JSONL viewer — reads session files, replays event stream hand by hand; requires PBI-1.5 stable event format |
| ICE-3 | Multiplayer — `Table` entity (`tableId`, `maxSeats`, `minBet`/`maxBet`, `numDecks`, `players`, `dealer`, `houseRules`), multi-player turn order (seat order then dealer), shared shoe (single or multi-deck), independent wallets per player. Session scoped to table lifetime — one JSONL file per table. Natural termination: last player leaves when broke. Forced termination: hall-close signal, graceful shutdown after current hand completes. Optional `max_hands` limit (default unlimited — valid in tournaments, promotional tables, and test/dev). New events: `TableOpened`, `TableClosed`, `PlayerSeated`. `PlayerJoined` deferred to ICE-8. `Player` gains `vip: bool` (default `False`). `HouseRules` dataclass introduced (ICE-7 compatible). Specced in TPS Section 10. |
| ICE-4 | Casino operations dashboard — real-time JSONL event aggregator that watches `logs/`, tails all active session files, and displays live casino-level statistics: active sessions, total hands played, house P&L, player win/loss/push rates, bust rates (player vs dealer), blackjack frequency, wallet trajectories per active session, and a live feed of recent hand outcomes. Requires PBI-1.5 stable event format. Builds on but is independent from ICE-2 (single-session replay). |
| ICE-5 | ML player strategy — train a classifier on simulated session JSONL data (hand state → action → outcome), wrap inference in the `strategy: Callable[[Hand], str]` interface, and plug into `Player` as a drop-in replacement for the deterministic strategy. Natural sequence: data generation → training → strategy plug-in → evaluation against deterministic baseline. Requires PBI-1.5 stable event format for training data collection. |
| ICE-6 | Human player strategy — interactive CLI strategy that blocks on `input()` for hit/stand decisions, surfacing the player's hand value and dealer upcard to the human before each decision. Requires extending the strategy interface from `Callable[[Hand], str]` to `Callable[[Hand, Card], str]` to pass the dealer upcard into the strategy function. Plug-in point is the existing `Player.strategy` callable — no changes to `play_hand()` or `play_session()` beyond the signature extension. |
| ICE-7 | Extended table rules — implements the remaining standard blackjack rule set as a bundle: (1) **Double down** — player doubles bet after initial deal, receives exactly one more card then stands; (2) **Split** — if first two cards share identical value, split into two independent hands each with their own bet; (3) **Insurance** — side bet of up to half the original bet when dealer shows Ace, pays 2:1 if dealer has blackjack; (4) **Surrender** — player folds before playing, recovers half the bet (late surrender — after dealer checks for blackjack); (5) **Soft 17 rule** — configurable dealer behaviour on soft 17 (hit or stand), added to `houseRules`; (6) **Blackjack payout ratio** — configurable 3:2 or 6:5, added to `houseRules`. Requires new events: `DoubleDown`, `SplitTaken`, `InsuranceTaken`, `InsurancePaid`, `SurrenderTaken`. All rules configurable via `houseRules` so they can be toggled per table. |
| ICE-8 | Floor manager (FM) — manages two player arrival queues: standard and VIP. VIP queue has priority; a configurable starvation threshold ensures standard queue players are not indefinitely skipped. FM performs table matching (on `minBet`/`maxBet`/`numDecks`/`houseRules`/available seats), creates a new table when no match exists, and handles player offer/accept/decline flow. FM is pluggable: machine mode (rules-based, fully automatic) or human mode (defers decisions to ICE-9 operator UI). Supports mid-session player join via `PlayerJoined` event. `Player.vip` flag determines queue assignment. Requires ICE-3. |
| ICE-9 | Floor manager UI — human operator mode for the FM. Extends the ICE-4 casino operations dashboard with operator controls: view standard and VIP player queues, review and action table offers, seat or decline players, open/close tables manually, trigger graceful hall shutdown. FM pluggable interface (ICE-8) routes to this UI when in human mode. Requires ICE-4 and ICE-8. |
| ICE-10 | Multi-seat play — one player occupying multiple seats, playing multiple simultaneous hands. Requires ICE-3. |
| ICE-11 | Back-betting — non-seated player places a side bet on a seated player's hand; seated player retains hit/stand authority. New events: BackBetPlaced, BackBetPaid. Requires ICE-10. |
| ICE-12 | Blackjack payout rounding — when minimum bet doesn't divide cleanly into the configured payout ratio, define rounding policy (round down, round up, even-money offer, 50-cent chip). Relevant when ICE-7 introduces variable bet sizes and configurable payout ratios. Requires ICE-7. |
| ICE-13 | Player identifier — add playerId: str (UUID4) to Player dataclass; key internal session state by playerId rather than name. Should precede ICE-8 (floor manager uses player identity). |
| ICE-14 | Event emission delay — configurable delay between events for human-readable dashboard playback; `emit_delay_ms: int \| None = None` parameter on `play_table_session()` and `play_session()`; disabled by default so tests are unaffected; target range 200–500ms. |
| ICE-15 | Session statistics and summary entities — (a) PlayerSession and TableSession structured summary entities: TableSession holds sessionId, tableId, handsPlayed, startedAt, closedAt, terminationReason, housePnl; PlayerSession holds sessionId, playerId, playerName, startingWallet, finalWallet, handsPlayed, stats. Supersedes finalWallet in SessionClosed. (b) Per-player and house-level running stats updated after each hand: hands played, wins, losses (non-bust), busts, pushes, blackjacks, P&L (cumulative and per-hand). Available during session, not just post-hoc. Feeds ICE-4 and ICE-9. PlayerSession is prerequisite for stats tracking. |
| ICE-16 | Dealer tipping — players can tip the dealer outside normal bet/payout flow; Dealer gains tips: float field; new TipGiven event (actor: player, data: amount, recipient: dealer); tips excluded from house P&L, tracked separately in session stats. Requires ICE-3. |
| ICE-17 | Voluntary player departure — player chooses to leave the table between hands with remaining funds; PlayerLeft with reason: "voluntary". Requires a mechanism to signal intent (likely via FM in ICE-8 or a future strategy interface extension). Requires ICE-8. |
| ICE-18 | Auto-evict on consecutive sit-outs — configurable house rule that removes a player after N consecutive SitOut events; N configurable per table via houseRules; fires PlayerLeft with reason: "sit-out limit reached". Requires ICE-20 (SitOut behaviour) and ICE-17. |
| ICE-19 | PlayerBooted behaviour — FM removes a player from the table or hall (event already reserved in TPS Section 9); spec the trigger conditions, FM authority rules, and whether booted player can re-enter. Requires ICE-8. |
| ICE-20 | SitOut behaviour — player skips a hand without placing a bet (event already reserved in TPS Section 9); spec the flow, wallet impact (none), and interaction with auto-evict rule (ICE-18). Prerequisite for ICE-18. |
